# Cedar Analysis MCP Tool: Verify and Analyze Cedar Authorization Policies

Authors: Spencer Erickson, Sr. Product Manager Technical, AWS & Liana Hadarean, Principal Applied Scientist, AWS

*June 16, 2025*

Today, we're excited to share a Cedar Analysis MCP tool that helps developers confidently create and modify Cedar authorization policies. Whether writing policies directly or using AI assistants to help author policies, teams need assurance that their policies work as intended. This MCP tool uses Cedar's recently launched Cedar Analysis capabilities to enhance the policy development process by checking policies and catching potential issues that might be overlooked. Using Cedar Analysis, developers can compare policy sets for unintended changes, detect conflicting or redundant rules, and verify policies maintain expected behavior across all possible scenarios. 

Consider, a collaborative photo sharing application where users can upload photos, manage privacy settings, and share content with colleagues. Each user belongs to a company, can own photos, and needs to share content within their company. Authorization in such applications quickly becomes complex, especially when implementing features like company-wide sharing and granular privacy controls.

Creating effective authorization policies often involves subtle nuances that even experienced developers can miss. For example, imagine you want to implement a policy allowing employees to view company-confidential photos shared within their company. This could be useful for sharing project updates or product designs. You might ask an AI assistant to help draft this policy, receiving this suggestion:

```cedar
// Allow users to view company private photos from anyone within their company
permit (
    principal is PhotoSharing::User,
    action == PhotoSharing::Action::"view",
    resource is PhotoSharing::Photo 
    ) 
    when { (principal == resource.owner) && (resource.companyPrivate == true) };
```

While this policy looks reasonable, it contains a critical flaw: it only allows owners to view their own private photos, rather than enabling company-wide sharing. The condition `(principal == resource.owner)` restricts access to owners, when it should use `(principal.company == resource.owner.company)` to allow sharing between colleagues. Without proper verification, such subtle logical errors could make it into production, leading to frustrated users and potential security risks.

The Cedar Analysis MCP tool addresses these challenges by verifying all policy changes, whether written by human developers or generated by AI assistants. It checks for logical flaws, unintended permissions, and potential security gaps that might otherwise go unnoticed. This creates a workflow where you can confidently develop and modify policies while ensuring they align with your requirements and existing policy set.

## Policy Verification Walkthrough Using Cedar Analysis MCP Tool

Let's see how this works in practice using the same photo sharing application. We'll demonstrate how you can work with Amazon Q Developer in your IDE to safely evolve your authorization policies. In this walkthrough, the developer has configured the Cedar Analysis MCP tool as an extension to Amazon Q Developer, though the same capabilities could be used with any coding assistant that supports the Model Context Protocol (MCP).

The walkthrough demonstrates a typical policy development workflow:

1. First,  we'll provide Amazon Q with context about our existing schema and policies
2. Then,  we'll attempt to add the company-wide photo sharing policy we discussed  earlier
3. Using  the Cedar Analysis MCP tool, Amazon Q will help us identify and fix issues  in real-time
4. Finally,  we'll verify the corrected policy behaves as intended

This interactive process shows how developers can combine the efficiency of AI-assisted coding with the rigorous verification capabilities of Cedar Analysis. Let's walk through each step in detail.

####  STEP 1: Provide Context to the AI Assistant 

 To begin our policy development process, we need to provide Amazon Q with the necessary context about our PhotoSharing application. This includes the existing schema and current policies. Let's examine what we're starting with: 

**User**: “I have a set of policies and Cedar schema defined in the `current_policies.cedar` file. Could you read the schema and my current policy set? I want to create new policies in this chat.”

**Amazon Q:** Certainly! I've analyzed your Cedar schema and current policies. Here's a summary:

Schema (`photo_sharing.cedarschema`): 

* Defines two entity types: Photo and User 
* Photo attributes: filename, owner, companyPrivate
* User attributes: company, email, isAdmin 
* Actions: comment, create, delete, view

Current Policies (`current_policies.cedar`):

1. Users can view any non-private photos
2. Users can view, comment on, and delete photos they own
3. Any user can create photos

**Note**: I’m sharing content of `current_policies.cedar` file for clarity to the reader of this blog post. 

```cedar
// Basic policies for PhotoSharing application
// 1. Users can view any non-private photos
@id("view-public-photos")
permit (
    principal is PhotoSharing::User,
    action == PhotoSharing::Action::"view",
    resource is PhotoSharing::Photo
)
when { resource.companyPrivate == false };

// 2. Users can view, comment on, and delete photos they own
@id("owner-full-access")
permit (
    principal is PhotoSharing::User,
    action in [PhotoSharing::Action::"view", PhotoSharing::Action::"comment", PhotoSharing::Action::"delete"],
    resource is PhotoSharing::Photo
)
when { resource.owner == principal };

// 3. Any user can create photos
@id("create-photos")
permit (
    principal is PhotoSharing::User,
    action == PhotoSharing::Action::"create",
    resource is PhotoSharing::Photo
);
```

User Schema

```json
namespace PhotoSharing {
    entity Photo = {
        "filename": String,
        "owner": User,
        "companyPrivate": Bool
    };
    entity User = {
        "company": String,
        "email": String,
        "isAdmin": Bool
    };
    // Additional schema elements...
}
```

### Step 2: User Wants to Verify a New Policy

**User:** “I want to add a new policy, defined below, that allows users from the same company to view each other's private photos. Could you verify that this new policy does not grant any unintended permissions?”

```cedar
@id("same-company-private-photo-access")
permit (
    principal is PhotoSharing::User,
    action == PhotoSharing::Action::"view",
    resource is PhotoSharing::Photo
)
when {
    (principal == resource.owner) &&
    (resource.companyPrivate == true)
};
```

### Step 3: Analysis via Cedar Analysis MCP Tool

**Amazon Q** calls the Cedar Analysis MCP tool to compare the policy sets:

```json
{
  "function": "compare-policy-sets",
  "parameters": {
    "policy_set1": "// Allow users to view resources they own...",
    "policy_set2": "// Allow users to view resources they own... + new policy...",
    "schema": "namespace PhotoSharing { entity Photo = {...} ..."
  }
}
```

### Step 4: Tool Returns Results and Identifies an Issue

**Amazon Q:** "I've analyzed the policy and compared it with your current policies. Here's what I found:

1. Policy Comparison Results: The comparison shows that all action types (comment, create, delete, view) are Equivalent between the two policy sets. This means the new policy doesn't grant any additional permissions beyond what's already allowed by your existing policies.
2. Policy Analysis: The analysis of the new policy shows no vacuous policies (policies that never apply) or other issues.
3. Logical Analysis: Looking at the policy itself: This policy permits a user to view a photo when the user is the owner of the photo AND the photo is private. However, this permission is already covered by your existing "owner-full-access" policy, which allows owners to view, comment on, and delete their own photos regardless of whether they're private or not.

Conclusion: The new policy is redundant and doesn't grant any new permissions. It's essentially a subset of your existing "owner-full-access" policy, focusing specifically on owners viewing their own private photos.

### Step 5: User Updates the Policy

**User:** “Here is an updated version of the policy, could you verify this time the policy behaves as intended?”

```cedar
// Policy to allow users from the same company to view each other's private photos
@id("same-company-private-photo-access")
permit (
    principal is PhotoSharing::User,
    action == PhotoSharing::Action::"view",
    resource is PhotoSharing::Photo
)
when {
    // User is from the same company as the photo owner AND the photo is private
    (principal.company == resource.owner.company) &&
    (resource.companyPrivate == true)
};
```

### Step 6: Final Verification

**Amazon Q** runs the analysis again:

```json
{
  "function": "compare-policy-sets",
  "parameters": {
    "policy_set1": "// Basic policies...",
    "policy_set2": "// Basic policies... + updated new policy...",
    "schema": "namespace PhotoSharing { entity Photo = {...} ..."
  }
}
```

**Amazon Q:** I've analyzed your updated policy. The policy now correctly implements the behavior you want - allowing users from the same company to view each other's private photos.

To verify this works as expected in practice, let's consider a few scenarios:

* User A  (company: "Amazon") tries to view a private photo owned by User  B (company: "Amazon") 
    * Result: ALLOW - Users from the same company can view each other's private photos
* User A (company: "Amazon") tries to view a private photo owned by User C (company: "ACME")
    * Result: DENY - Users from different companies cannot view each other's private photos
* User A (company: "Amazon") tries to view a non-private photo owned by User C (company: "ACME")
    * Result: ALLOW - Anyone can view non-private photos (from the view-public-photos policy)
* User A (company: "Amazon") tries to delete a photo owned by User B (company: "Amazon")
    * Result: DENY - Same company access only applies to viewing, not deleting

The policy now behaves as intended, allowing company-based sharing of private photos while maintaining all other existing permissions.

## Cedar Analysis with MCP

The Cedar Analysis MCP Tool connects Cedar's verification capabilities to AI assistants through the Model Context Protocol (MCP). This standardized protocol allows AI systems like Amazon Q to access Cedar's powerful policy analysis functions during conversations with developers.

The tool runs policy verification behind the scenes while you interact with Amazon Q, providing immediate feedback on policy changes without disrupting your workflow. This creates a seamless experience where you can discuss, generate, and verify policy changes all within the same conversation.

Cedar provides formal verification capabilities that validate security properties across all possible scenarios. Our new tools use this capability to verify both human and LLM generated policies, enabling you to:

* Move quickly while maintaining strong confidence in your security controls
* Validate that policies implement your intended permissions
* Understand how policy changes affect your security posture

The Cedar Analysis MCP tool exposes two primary functions that leverage Cedar's policy verification capabilities. Learn more in our [Cedar Analysis blog post](https://www.cedarpolicy.com/blog/introducing-cedar-analysis)

1. **Analyze**: Examines a single policy set to identify logical errors, inconsistencies, conflicts, and redundant rules. This helps teams catch potential security issues and optimize their policies.
2. **Compare**: Evaluates two policy sets to determine if they grant equivalent permissions and understand the impact of policy changes. This enables teams to confidently update their authorization rules while avoiding unintended access modifications.

## Getting Started

To get started you can download and install the MCP server for Cedar analysis locally following the instructions [here](https://github.com/cedar-policy/cedar-examples/blob/release/4.4.x/cedar-analysis-mcp-server/README.md). Then simply add the following to your MCP host application configuration e.g. for Amazon Q this would be `~/.aws/amazonq/mcp.json`:

```json
{
  "mcpServers": {
    "cedar-cli": {
      "command": "docker",
      "args": ["run", "-i", "--rm", "cedar-cli"],
      "env": {},
      "disabled": false,
      "autoApprove": []
    }
  }
}
```

 The Cedar Analysis MCP server extends any MCP-compatible coding assistant with Cedar policy analysis tools, allowing you to validate, test, and analyze Cedar authorization policies directly in your development workflow. Whether you're using Amazon Q, Claude, or any other assistant that supports MCP, you can leverage Cedar's verification capabilities within your preferred development environment.
